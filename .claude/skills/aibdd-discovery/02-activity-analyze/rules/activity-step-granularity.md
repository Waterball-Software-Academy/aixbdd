# Step（Action）顆粒度

- **Activity 不是循序圖**：同一次使用者業務意圖，**不得**拆成多個並列 Action 以表示內部「系統計算 → 寫庫 → 呼叫第三方」等技術微步。
- **一個 Action**＝**一次**由符合 **Actor 顆粒度**之觸發者所觸發、且使用者能感知**完成結果**的**完整業務行動**。
- 跨產品或第三方 boundary 之後果，**以 postcondition 或對應 feature 內規則敘述表達**，**不得**另立僅代表內建系統處理之獨立 Action。
- **Action 名稱**表**業務意圖**，**不得**等同實作細節或內部技術步驟名稱。
- **每個 Action** 在概念上 **binds_feature** 恰對 **一個** feature（矩形）；草案階段路徑可標 gap，一對一關係不變。
- **API-Like Verification**: 你可以用 API 去想像一個 Action 的顆粒度，我們應盡可能確保每一個 Step(Action)/Feature File 都像良好的 API 設計一樣具備底下品質：
    1. 狀態轉換原則
    - [ ] API 是否自行處理所有狀態轉換邏輯？
    - [ ] 是否避免設計專門用於狀態轉換的 API？（如：/set-status 這類 API 不應存在）
    - [ ] 狀態轉換是否作為其他 API 操作的副作用自動完成？
    2. 業務邏輯封裝原則
    - [ ] 相關的業務邏輯是否封裝在同一個 API 中？（避免一個業務動作拆成多個 API）
    - [ ] 驗證、計算、更新等相關操作是否在同一個 API 內完成？
    - [ ] 是否避免出現「流程分解過細」的 API 設計？
    3. 前端職責原則
    - [ ] 前端是否僅負責 UI 呈現與輸入驗證？
    - [ ] 是否避免設計專門用於前端狀態管理的 API？
    - [ ] 是否減少前端對後端的輪詢需求？（如有即時需求，是否考慮 WebSocket 或推播）
- **Distinct operation**：若敘述已區分多個可**獨立驗收、獨立命名或獨立觸發**之 operation，**不得**共用同一 **binds_feature**；應對應**多個 Action**，各綁**不同** feature。
- **command** 與 **query**：具寫入／副作用者與純查詢者**分屬不同** feature；command 類 Action **不得**綁定僅表述 query 之 feature，反之亦然。

# 反例

- 同一次「業務人員送出客戶評級」意圖，拆成多個 Action：**送出表單**、**系統計算風險**、**寫入 CRM**、**第三方寄信**——把內部技術鏈當成並列步驟。預期應**收斂為單一 Action**（由該業務人員觸發、名稱表整段業務意圖），風險已計算、CRM 已更新、通知已排程或已寄送等後果，**只**出現在 **postcondition** 或對應 **feature** 之規則敘述，**不**另立系統／第三方 Action。
- Action 命名成 **`callPaymentGateway`**、**`updateRowInDb`** 等實作或介面名稱，而非可讀的業務意圖。
- **建立訂單**與**取消訂單**兩種可獨立驗收、獨立觸發的 operation，卻共用同一 **binds_feature**／同一顆矩形。
- **下單（會改寫狀態）**綁在僅描述 **查詢訂單列表** 的 feature，或把純 **查詢儀表板**綁在有副作用的付款 feature。
